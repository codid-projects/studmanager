Frontend Breeding Changes

Complete frontend handoff · 20 June 2026

TypeScript and production build verified

Summary

The legacy breeding interface was replaced with API-backed mare and stallion workspaces based on the supplied Figma and screenshots. The implementation supports Arabic/English, RTL/LTR, responsive layouts, custom horse selection, forms, record tables, and breeding calendar events.

Custom horse pickers

  • Removed native browser horse dropdowns from breeding forms.
  • The main mare/stallion selector, sire selector, donor mare selector, and surrogate mare selectors use one shared custom modal.
  • Each modal requests internal horses from the backend using the correct male/female filter.
  • Shows 10 horses per page with a fixed responsive modal height and scrollable result area.
  • Arabic and English name search is debounced; searching returns to page 1.
  • Previous/next pagination uses backend total counts.
  • Selecting a horse closes the modal automatically and displays the selected name in a bordered field.
  • Optional horse inputs include a clear button.
  • The active donor/current mare is disabled when choosing a surrogate, preventing duplicate selection.
  • Small fixed lists such as gender, status, method, storage type, and clinical result remain compact native controls.

Date and calendar input behavior

  • Date, datetime and month controls retain a visible calendar icon.
  • Clicking anywhere in the input opens its picker, not only the icon.
  • Breeding calendar types are localized in English and Arabic.
  • Changing an event type automatically selects its readable pastel color; manual color override remains available.

API integration

  • Separate mare and stallion breeding API clients.
  • Gender-filtered internal-horse paging and search.
  • Profile resolution/creation after selecting a horse.
  • API-backed create, list, refresh, and delete flows.
  • Forms reset both native controls and controlled horse selections after successful saves.
  • Arabic and English API errors are presented inside the active workflow.

Save reset defect fixed

The runtime error Cannot read properties of null (reading 'reset') was fixed in every asynchronous breeding form. Each handler now stores the form element before awaiting the API request and resets that stable reference after a successful save.

Affected workflows: ovulation examination, mare soundness, estrus cycle, semen collection, semen shipment and stallion soundness. Natural breeding and foal registration already use the same safe pattern.

Horse sale workflow

  • The profile action opens a sale modal instead of immediately toggling the status.
  • New sales require two string values: buyer and displayed sale price.
  • After saving, opening the action previews buyer, price and sale date.
  • Existing sale details can be edited in the same modal.
  • Remove sale clears buyer, price, timestamp and sold state, returning the horse to available.
  • The modal supports Arabic/English, loading, validation and API error states.

Stallion workflow interaction update

  • Natural breeding, semen collection, semen shipment and soundness creation forms now start collapsed inside a clear expandable action card.
  • Saving, loading record details, refreshing lists and deleting display visible loading indicators and disable conflicting actions.
  • Every stallion record can be edited in a modal populated from the backend. Natural-breeding mare fields use the searchable custom horse picker.
  • Deleting opens a bilingual confirmation modal and cannot be triggered accidentally from the table.
  • Storage types, collection methods and the remaining enum choices are translated into Arabic while preserving their API enum values.
  • Mare and stallion workspaces remember separate selected horses. Switching gender never carries an invalid horse into the other workspace.
  • Selection checkboxes and the unused multi-select column were removed from all breeding record tables.

Mare card and finance update

  • New foal, breeding control, estrus-cycle tracking and mare soundness forms now start collapsed, keeping their record tables visible in the first viewport.
  • Every price control added or reused in breeding is labelled as EGP and serializes into the backend billed-service format.
  • The expenses area has a new Breeding category backed by live ledger records rather than mock rows.
  • The revenue screen now reads live breeding-related revenue, including horse sales, rather than displaying mock invoices.
  • Both financial views support loading, empty/error states, backend pagination and a detail modal showing subtype, horse, description, account lines and total.
  • Arabic/English labels are supplied for every breeding financial subtype.
